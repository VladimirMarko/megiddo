# Separate Identity Auth Provider, Token Codec, And Browser Session Concerns

Identity will configure authentication and token cryptography through separate modes. `IDENTITY_AUTH_PROVIDER=dummy | better-auth` selects the Auth Provider Adapter, while `IDENTITY_TOKEN_CODEC=dummy | jwt-jws` selects the Identity Token signer/verifier implementation. `MEGIDDO_AUTH_PROFILE=local-dummy` is a local convenience preset that enables dummy auth, seeded dummy demo accounts, dummy token codec, and the frontend dummy-auth login shortcut, but each setting remains independently overrideable. Dummy auth provides persistent accounts with no meaningful credential security: anyone may sign in as an existing dummy principal, but sign-in never creates principals implicitly. `IDENTITY_DUMMY_AUTH_DEMO_ACCOUNTS=enabled` seeds normal persisted dummy accounts for Alice and Bob; later, a seed file can replace or extend that list. `UI_DUMMY_AUTH_LOGIN_SHORTCUT=enabled` is a frontend concern that renders the available dummy principals as one-click sign-in options, not an Identity or Gateway policy. Dummy auth and dummy tokens are refused when `NODE_ENV=production`.

Better Auth will live inside the Identity service for the MVP. Browser login uses Identity-owned session cookies in both Better Auth and dummy auth modes, and the normal frontend path does not receive Megiddo service tokens. The frontend talks only to API Gateway; Gateway forwards browser session context to Identity to resolve the current session and asks Identity to issue downstream service tokens server-side. The normal browser path no longer needs an `api-gateway` audience token. Service-token issuance is internal: the first slice may authenticate Gateway to Identity with a shared internal service secret, but Identity must also enforce an audience allowlist such as `api-gateway -> todo` rather than exposing a public mint-any-audience endpoint. The dummy token codec uses explicit unsigned inspectable tokens shaped as `dummy.<base64url-json-claims>`; the real codec should be standards-based JWT/JWS. Once `dummy` and `jwt-jws` exist, the current custom Ed25519 compact token format should not remain as a long-term supported codec.
