# Architecture History Source Inventory

Last updated: 2026-07-08.

This inventory maps existing source material to Megiddo's development-history and architecture-rationale narrative. It is not the final narrative. Use it to decide which claims are supported, which sources are canonical, and which historical notes are transitional.

ADRs remain the canonical source for individual architecture decisions. Reports, README sections, the env catalog, and issue history provide supporting context and implementation chronology.

- The local OpenTelemetry viewer report's rejected candidates are historical evaluation evidence. The current selected viewer is `otel-gui` per the report recommendation and README workflow.
- [`ADR-0028`](../adr/0028-use-fly-for-the-first-production-shaped-staging-deployment.md) is a temporary first-provider decision. Do not describe Fly as a permanent platform commitment.
- [`ADR-0026`](../adr/0026-use-production-shaped-staging-on-fly-with-service-owned-sqlite-volumes.md), [`ADR-0030`](../adr/0030-start-staging-observability-with-platform-logs-and-health-checks.md), and [`ADR-0032`](../adr/0032-defer-database-migrations-for-first-staging-deployment.md) intentionally preserve first-staging limitations: single-writer SQLite volumes, platform logs and health checks only, no database migrations, no backups, and no horizontal scaling for stateful Services.

## Source Gaps for Later Writing

- Frontend stack rationale is thin. [`ADR-0017`](../adr/0017-use-react-vite-tanstack-and-jotai-for-frontend.md) names React, Vite, TanStack Router/Form, and Jotai, but does not provide detailed rationale for those choices. State only the documented rationale unless issue #1 or implementation history provides more.
- Service ownership is mostly vocabulary and implementation shape, not a dedicated ADR. Use `CONTEXT.md`, ADR-0002, ADR-0011, package boundaries, and early issue history carefully.
- Runtime support for multiple live contract versions is policy-backed by ADR-0009 and ADR-0013. Verify current implementation before claiming a specific service currently serves more than one live version.
- Current Better Auth behavior should be verified against code or issue #35 before the final narrative makes detailed claims. The strongest durable source is ADR-0024, while the identity report predates later identity slices.
- First staging deployment artifacts and runbooks are repo-owned preparation, not evidence that a live Fly deployment has been performed. Use the first-live-deploy handoff only as an operator checklist until deployment evidence is recorded.
- Exact chronological ordering beyond ADR number, report date, and issue number is approximate. PRD #49 asks for a supported approximate timeline, not a perfect commit-by-commit history.

## Recommended Narrative Source Order

1. Start from the ADR sequence for the decision index and timeline spine.
2. Use README for current topology, local development workflow, telemetry viewer workflow, env-loading policy, and testing commands.
3. Use `CONTEXT.md` for terminology and to keep vocabulary consistent.
4. Use env catalog for current environment details and the environment report only as a pre-migration snapshot.
5. Use the identity report only as historical baseline and gap analysis, then continue through ADR-0024 and issues #27 through #35.
6. Use the local telemetry viewer report as evidence for choosing `otel-gui` and for not building custom devtools first.
7. Use ADR-0026 through ADR-0032 for staging deployment rationale, and use the staging runbook for operator procedure.
8. Use issue clusters to explain why work arrived in slices, especially where the ADRs are terse.
